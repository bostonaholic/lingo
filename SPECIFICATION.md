# The Lingo Programming Language Specification

**Version:** 0.1.0 (Draft)
**Date:** 2026-03-15

---

## Executive Summary

Lingo is a general-purpose programming language designed from first principles to minimize the
number of tokens an LLM must generate to express correct, idiomatic programs -- without degrading
the LLM's ability to reason about, debug, or modify that code.

The core insight driving Lingo's design is empirical: the largest token savings in programming
language design come not from making code shorter, but from eliminating tokens that carry zero
information. Type annotations the compiler could infer, boilerplate delimiters that could be
structural, error-handling ceremony that could be a single operator, nested call syntax that
could be linear -- these are the tokens Lingo eliminates.

Lingo targets **90-110 average tokens** per Rosetta Code task, placing it between J (~70 tokens,
too terse for reliable LLM reasoning) and Python (~130 tokens, the LLM reasoning baseline). It
achieves this through Hindley-Milner type inference, first-class pipeline composition, Rust-style
error propagation, and algebraic data types -- while using syntax familiar to models trained on
Python, JavaScript, Rust, and Haskell.

**Target domain:** Data transformation pipelines, CLI tools, and backend services -- the programs
most frequently generated by LLMs in coding assistants.

**File extension:** `.ln`

## 1. Design Philosophy

Lingo is governed by five principles, each grounded in empirical research on LLM token efficiency
and code reasoning quality.

### Principle 1: Eliminate Zero-Information Tokens

> Every token in a Lingo program must carry semantic information that neither the compiler nor the
> LLM could infer from context.

Formatting overhead alone consumes 25-35% of tokens in brace-delimited languages like Java (34.9%)
and C++ (31.1%). Type annotations that a Hindley-Milner inference engine can derive, boilerplate
class hierarchies that algebraic data types replace in a single declaration, error-handling ceremony
that a single `?` operator subsumes -- these are zero-information tokens that Lingo eliminates.

### Principle 2: Preserve Semantic Anchors

> Structural compression is safe; semantic compression is not.

Research demonstrates that removing formatting has near-zero impact on state-of-the-art LLM
reasoning (GPT-4o, DeepSeek-V3, and Claude show negligible degradation from whitespace removal).
In stark contrast, anonymizing identifier names causes up to 75% performance collapse on code
search tasks. Descriptive identifiers cost 41% more tokens but achieve 2x the task performance.
Lingo aggressively compresses structural tokens while preserving and encouraging descriptive
identifiers, explicit control flow, and clear function boundaries.

### Principle 3: Familiar Syntax, Novel Efficiency

> The less novel the syntax, the less training data is needed.

Lingo starts with zero training data. Every keyword and operator is drawn from the intersection
of conventions LLMs already know: `fn` (Rust), `let` (JavaScript/Rust), `|>` (Elixir/F#),
`match` (Rust), `=>` (JavaScript/Scala). Research on novel DSLs (DSL-Xpert, MODELS 2024)
demonstrated that LLMs can generate reliable code for unpublished languages when given grammar
specifications as context -- meaning Lingo's grammar serves as its own bootstrap mechanism. The
grammar is compact enough (~200 lines of PEG) to fit in a prompt alongside a task description.

### Principle 4: Types Reduce Total Cost

> Type annotations cost tokens, but types reduce the iteration tax.

Typed languages produce more verbose first drafts, but type checkers catch errors before they
become multi-turn debugging sessions. The net token cost across a full generate-debug cycle is
lower with types than without. Hindley-Milner inference resolves the tension between type safety
and verbosity: Haskell (115 average tokens) and F# (118) rival Python (130) despite being
statically typed. Lingo achieves dynamic-language token density with static-language correctness
guarantees.

### Principle 5: ASCII Only

> Current BPE tokenizers are optimized for ASCII code.

The APL-vs-J comparison is definitive: APL's Unicode glyphs are tokenized as 1-3 tokens each by
BPE tokenizers trained on code corpora, while J's ASCII equivalents are single tokens. Every
Lingo operator, keyword, and delimiter uses only ASCII characters. Keywords are chosen to be
single tokens in major tokenizer vocabularies (cl100k_base, o200k_base).

### Key Tensions and Resolution Strategy

**Conciseness vs. Ambiguity.**
Each operator has exactly one meaning regardless of context. Token savings come from eliminating
redundant structure, not from overloading symbols. J's 70-token average relies on
context-dependent operators that LLMs cannot reason about reliably.

**Brevity vs. Learnability.**
Syntax is a strict subset of conventions already known to LLMs from Rust, Haskell, Elixir, and
JavaScript. The grammar is compact enough for prompt-based generation from day one.

**Type Safety vs. Token Cost.**
Hindley-Milner inference within function bodies, with annotations required only at
module-boundary function signatures where they serve as documentation and LLM reasoning anchors.

## 2. Target Domain

### Primary Niche: Data Transformation and CLI Tooling

Lingo targets the domain where token-efficient design delivers the highest marginal value:
**data transformation pipelines, CLI tools, and backend services**.

**Justification:**

1. **LLMs generate Python for 90-97% of coding tasks.** The dominant use cases are scripting,
   data manipulation, and service endpoints. Lingo competes directly with Python on these tasks
   while delivering 15-30% fewer tokens with static type safety.

2. **Pipeline composition is the highest-value syntax feature.** Data transformation is inherently
   sequential: read, parse, transform, filter, aggregate, output. The `|>` operator maps directly
   to this workflow, eliminating nested parentheses and intermediate variables.

3. **Error handling dominates real-world token cost.** Benchmarks show Python at 4,322 tokens vs.
   Rust at 6,064 tokens for a CLI task manager -- a 40% penalty largely attributable to Rust's
   explicit error handling. Lingo's `?` operator eliminates this overhead while preserving
   explicit fallibility markers in function signatures.

4. **Real-project token counts matter more than micro-benchmarks.** Real projects include
   configuration files and boilerplate. Lingo's module system minimizes import ceremony, and its
   rich built-in namespace eliminates boilerplate for common I/O, string, and collection
   operations.

5. **CLI tools and data pipelines are the sweet spot for grammar-prompted generation.** These
   programs are typically 50-200 lines, well within the context window budget for including
   Lingo's grammar alongside the task description.

## 3. Language Spine

### 3.1 Lexical Conventions and Syntax

#### 3.1.1 Keywords

Lingo has 22 keywords. Every keyword is a single token in cl100k_base and o200k_base. Short
keywords are preferred because they produce fewer bytes in generated output.

| Keyword  | Purpose                     | Familiar From       |
| -------- | --------------------------- | ------------------- |
| `fn`     | Function declaration        | Rust                |
| `let`    | Immutable binding           | JS, Rust, Haskell   |
| `mut`    | Mutable binding modifier    | Rust                |
| `if`     | Conditional                 | Universal           |
| `else`   | Conditional branch          | Universal           |
| `match`  | Pattern matching            | Rust                |
| `for`    | Iteration                   | Universal           |
| `in`     | Iterator binding            | Python, Rust        |
| `while`  | Conditional loop            | Universal           |
| `loop`   | Infinite loop               | Rust                |
| `break`  | Loop exit                   | Universal           |
| `return` | Early return only           | Universal           |
| `type`   | Type alias / ADT definition | Haskell, TypeScript |
| `struct` | Record type                 | Rust                |
| `enum`   | Sum type                    | Rust                |
| `trait`  | Interface / typeclass       | Rust                |
| `impl`   | Implementation block        | Rust                |
| `mod`    | Module declaration          | Rust                |
| `use`    | Import                      | Rust                |
| `pub`    | Public visibility           | Rust                |
| `async`  | Async function modifier     | JS, Rust            |
| `await`  | Await expression            | JS, Rust            |

**Rejected keywords and why:**

- `class` -- replaced by `struct` + `trait`, which compose more flexibly
- `try` / `catch` / `throw` -- replaced by `Result` + `?` operator
- `var` -- replaced by `let mut` for explicit mutability
- `null` / `nil` -- replaced by `Option` type; null is eliminated entirely
- `do` / `end` -- block delimiters use braces for maximum LLM familiarity

#### 3.1.2 Operators

Lingo has 28 operators. Each operator has exactly one meaning regardless of context -- no
overloading. All operators are ASCII and single tokens in cl100k_base.

| Operator | Category               | Meaning                |
| -------- | ---------------------- | ---------------------- |
| `+`      | Arithmetic             | Addition               |
| `-`      | Arithmetic             | Subtraction / negation |
| `*`      | Arithmetic             | Multiplication         |
| `/`      | Arithmetic             | Division               |
| `%`      | Arithmetic             | Remainder              |
| `==`     | Comparison             | Equality               |
| `!=`     | Comparison             | Inequality             |
| `<`      | Comparison             | Less than              |
| `>`      | Comparison             | Greater than           |
| `<=`     | Comparison             | Less than or equal     |
| `>=`     | Comparison             | Greater than or equal  |
| `&&`     | Logical                | Logical AND            |
| `\|\|`   | Logical                | Logical OR             |
| `!`      | Logical                | Logical NOT            |
| `=`      | Assignment and Binding | Binding / assignment   |
| `+=`     | Assignment and Binding | Add-assign             |
| `-=`     | Assignment and Binding | Subtract-assign        |
| `\|>`    | Composition and Access | Pipeline forward       |
| `=>`     | Composition and Access | Lambda / match arm     |
| `->`     | Composition and Access | Return type            |
| `.`      | Composition and Access | Field access           |
| `::`     | Composition and Access | Module path            |
| `..`     | Composition and Access | Range (exclusive)      |
| `..=`    | Composition and Access | Range (inclusive)      |
| `?`      | Error Handling         | Error propagation      |
| `:`      | Type Annotation        | Type annotation        |
| `\|`     | Type Annotation        | Variant separator      |

**Bitwise operations** are provided as named functions in the standard library (`bit_and`,
`bit_or`, `bit_xor`, `bit_not`, `shl`, `shr`) rather than symbolic operators. This avoids
overloading `&` and `|`, which would create ambiguity with `&&`, `||`, and the enum variant
separator `|`.

#### 3.1.3 Delimiters

| Token | Purpose                                          |
| ----- | ------------------------------------------------ |
| `{}`  | Block delimiters, struct/enum bodies             |
| `()`  | Grouping, function parameters, tuples            |
| `[]`  | Array/slice indexing, array literals, generics   |
| `,`   | Element separator                                |
| `;`   | Optional statement separator (newline works too) |
| `#`   | Line comment                                     |

**Block delimiter rationale.** Lingo uses braces because: (a) significant whitespace requires
LLMs to count indentation to understand structure, which is error-prone; (b) `end` keywords are
unambiguous but add a token per block; (c) braces are the most familiar delimiter to models
trained on JavaScript, Rust, C, and Java. The formatting overhead from braces is offset by
Lingo's savings in type inference, pipeline composition, the `?` operator, and implicit returns.

**Comment syntax.** `#` for line comments (single token, same as Python). No block comment
syntax -- multi-line comments use multiple `#` lines. This avoids the `/* */` token pair and
aligns with the Python convention that LLMs know best.

#### 3.1.4 Whitespace Rules

- **Newlines** are statement terminators (like Go). Semicolons are optional and equivalent to
  newlines.
- **Indentation** is not significant. Structure is determined by braces.
- **Blank lines** are ignored.
- **Continuation:** A newline is suppressed as a terminator after tokens that clearly continue
  an expression: `|>`, `=>`, binary operators, `(`, `[`, `{`, `,`.

#### 3.1.5 Identifier Conventions

| Convention             | Usage                           |
| ---------------------- | ------------------------------- |
| `snake_case`           | Functions, variables, modules   |
| `PascalCase`           | Types, structs, enums, traits   |
| `SCREAMING_SNAKE_CASE` | Constants                       |

Short names (`x`, `i`, `n`) are acceptable in narrow scopes: lambda parameters, pattern match
bindings, and loop variables. Descriptive names are required at function boundaries and
module-level definitions.

`snake_case` is the default convention because snake_case identifiers split more predictably in
BPE tokenizers than camelCase (the o200k_base pretokenizer introduces native camelCase splitting,
creating less regular subword segments).

#### 3.1.6 String Literals

Double-quoted strings support interpolation with `{expr}` inside the string. No prefix character
is needed (saving 1 token vs. Python's `f"..."`).

```lingo
let greeting = "Hello, {name}! You are {age} years old."
let summary = "Total: {items |> len} items, {total_cost} credits"
```

Escape literal braces with backslash: `"\{not interpolated\}"`.

Raw strings for regex and paths use an `r` prefix:

```lingo
let pattern = r"^\d{3}-\d{4}$"
```

Character literals use single quotes:

```lingo
let ch = 'A'
```

### 3.2 Type System

#### 3.2.1 Foundation: Hindley-Milner Inference

Lingo uses bidirectional type inference based on Hindley-Milner. This is the single
highest-impact design choice for token efficiency: it closes the gap between statically typed
and dynamically typed languages. Haskell (115 average tokens) and F# (118) rival Python (130)
despite full static typing, because type inference eliminates annotation tokens that are
redundant with the compiler's knowledge.

**Annotation rules:**

1. **Within function bodies:** All types are inferred. No annotations needed.
2. **Top-level function signatures:** Parameter and return type annotations are required. They
   serve as documentation and LLM reasoning anchors.
3. **Local `let` bindings:** Never need type annotations.
4. **Struct fields:** Always explicitly typed (they define the data model).
5. **Generic type parameters:** Inferred at call sites.

```lingo
# Types inferred within body; signature annotated at boundary
fn process(items: [Item], threshold: Float) -> [Item] {
  let filtered = items |> filter(i => i.value > threshold)
  let sorted = filtered |> sort_by(i => i.name)
  sorted
}
```

#### 3.2.2 Built-in Types

| Type     | Description               | Notes                              |
| -------- | ------------------------- | ---------------------------------- |
| `Int`    | 64-bit signed integer     | Short name; single token           |
| `Float`  | 64-bit floating point     | Short name; single token           |
| `Bool`   | Boolean                   | `true` and `false` are literals    |
| `Str`    | UTF-8 string              | Short for String; single token     |
| `Char`   | Unicode scalar value      | Single-quoted literal              |
| `[T]`    | List of T                 | Square bracket syntax              |
| `{K: V}` | Map from K to V           | Brace syntax with colon            |
| `{T}`    | Set of T                  | Brace syntax without colon         |
| `(A, B)` | Tuple                     | Parenthesized, comma-separated     |
| `()`     | Unit type                 | Empty tuple; used for side effects |

**Why short type names:** `Int` rather than `Integer`, `Str` rather than `String`, `Bool` rather
than `Boolean`. Both short and long forms are single tokens in cl100k_base, but shorter names
produce fewer bytes in generated output. Every saved byte reduces cost.

#### 3.2.3 Option and Result

Null is eliminated from the language entirely. All optional values use the `Option` type; all
fallible operations use the `Result` type. Both are built into the prelude and require no imports.

```lingo
# Option -- represents presence or absence
enum Option[T] {
  Some(T)
  None
}

# Result -- represents success or failure
enum Result[T, E] {
  Ok(T)
  Err(E)
}
```

The constructors `Some`, `None`, `Ok`, and `Err` are globally available.

#### 3.2.4 Sum Types (Algebraic Data Types)

Sum types express variant data in a fraction of the tokens required by class hierarchies. Adding
a variant becomes a compile error at every `match` site, making modifications mechanical and
LLM-friendly.

```lingo
enum Shape {
  Circle(radius: Float)
  Rect(width: Float, height: Float)
  Point
}

enum Color {
  Rgb(r: Int, g: Int, b: Int)
  Hex(value: Str)
  Named(name: Str)
}
```

#### 3.2.5 Struct Types

Structs define record types with named fields.

```lingo
struct User {
  name: Str
  age: Int
  email: Str
}
```

Instantiation uses named fields:

```lingo
let user = User { name: "Alice", age: 30, email: "alice@example.com" }
```

Field shorthand when a variable name matches the field name:

```lingo
let name = "Alice"
let age = 30
let email = "alice@example.com"
let user = User { name, age, email }
```

#### 3.2.6 Generics

Generics use square brackets (like Scala) to avoid ambiguity with comparison operators:

```lingo
fn first[T](items: [T]) -> Option[T] {
  match items {
    [head, ..] => Some(head)
    [] => None
  }
}

fn map_pair[A, B](pair: (A, A), f: (A) -> B) -> (B, B) {
  (f(pair.0), f(pair.1))
}
```

#### 3.2.7 Trait System

Traits define shared behavior. Lingo uses structural typing as the default: a type satisfies a
trait if it implements its methods, without requiring an explicit `impl Trait for Type`
declaration. Explicit `impl` blocks are available for documentation, disambiguation, and
organizing associated methods.

**Trait definition:**

```lingo
trait Display {
  fn display(self) -> Str
}

trait Numeric {
  fn zero() -> Self
  fn add(self, other: Self) -> Self
}
```

**Explicit implementation:**

```lingo
impl Display for User {
  fn display(self) -> Str {
    "{self.name} (age {self.age})"
  }
}
```

**Trait bounds on generics:**

```lingo
fn print_all[T: Display](items: [T]) {
  for item in items {
    println(item.display())
  }
}

fn sum[T: Numeric](items: [T]) -> T {
  items |> fold(T::zero(), (acc, x) => acc.add(x))
}
```

**Multiple trait bounds:**

```lingo
fn process[T: Display + Numeric](items: [T]) -> Str {
  let total = sum(items)
  total.display()
}
```

### 3.3 Control Flow and Pattern Matching

Everything in Lingo is an expression. `if/else`, `match`, and blocks all return values. This
eliminates temporary variables and declaration-then-assignment patterns.

#### 3.3.1 If / Else

`if/else` is an expression that returns a value.

```lingo
# Single-line (expression)
let status = if score >= 90 { "A" } else if score >= 80 { "B" } else { "C" }

# Multi-line
let result = if is_valid(input) {
  process(input)
} else {
  Err("invalid input")
}

# As a statement (unit return)
if should_log {
  println("Processing started")
}
```

#### 3.3.2 Match (Exhaustive Pattern Matching)

Pattern matching is the primary dispatch mechanism. The compiler enforces exhaustiveness --
unhandled variants are compile errors.

```lingo
fn area(shape: Shape) -> Float {
  match shape {
    Circle(r) => 3.14159 * r * r
    Rect(w, h) => w * h
    Point => 0.0
  }
}
```

**Supported pattern forms:**

| Pattern Kind | Example | Description |
| --- | --- | --- |
| Literal | `42`, `"hello"`, `true` | Matches exact values |
| Variable binding | `x`, `name` | Binds matched value to a name |
| Constructor | `Some(x)`, `Err(e)`, `Circle(r)` | Destructures enum variants |
| Tuple | `(a, b)`, `(0, _)` | Destructures tuples |
| List | `[first, ..rest]`, `[x, y]`, `[]` | Destructures lists with rest |
| Struct | `User { name, age, .. }` | Destructures struct fields |
| Wildcard | `_` | Matches anything, discards value |
| Guard | `Some(x) if x > 0 => ...` | Pattern with boolean condition |
| Or-pattern | `Circle(_)` or `Point => ...` | Matches any of several patterns |
| Range | `1..=10` | Matches values within a range |

**Complex match example:**

```lingo
fn describe(value: Option[Int]) -> Str {
  match value {
    Some(x) if x > 100 => "large: {x}"
    Some(x) if x > 0 => "positive: {x}"
    Some(0) => "zero"
    Some(x) => "negative: {x}"
    None => "absent"
  }
}
```

**Nested pattern matching:**

```lingo
fn process_result(r: Result[Option[Int], Str]) -> Str {
  match r {
    Ok(Some(n)) => "got {n}"
    Ok(None) => "empty success"
    Err(msg) => "error: {msg}"
  }
}
```

#### 3.3.3 For Loops

```lingo
# Basic iteration
for item in items {
  println(item.display())
}

# With index (via pipeline to enumerate)
for (i, item) in items |> enumerate {
  println("{i}: {item}")
}

# Over a range
for i in 0..10 {
  println(i)
}

# Inclusive range
for i in 0..=10 {
  println(i)
}
```

#### 3.3.4 While Loops

```lingo
while buffer |> len < max_size {
  let chunk = read_chunk()?
  buffer = buffer ++ chunk
}
```

#### 3.3.5 Loop (Infinite)

```lingo
loop {
  let line = read_line()?
  if line == "quit" { break }
  process(line)
}
```

`loop` is syntactic sugar for `while true` and communicates infinite-loop intent clearly.

### 3.4 Functions, Closures, and Composition

#### 3.4.1 Function Declaration

Top-level functions require type annotations at boundaries. The last expression in a function
body is the implicit return value. The `return` keyword is reserved for early exits only.

```lingo
fn add(a: Int, b: Int) -> Int {
  a + b
}

fn greet(name: Str) -> Str {
  "Hello, {name}!"
}

fn classify(score: Int) -> Str {
  if score < 0 { return "invalid" }
  if score >= 90 { "excellent" }
  else if score >= 70 { "good" }
  else { "needs improvement" }
}
```

**Implicit return convention:** The final expression in any block is its value. This follows the
Rust convention: no `return` at the end of a function, explicit `return` for early exit only.
This creates a consistent, learnable rule.

#### 3.4.2 Lambda Syntax

Arrow lambda syntax is universally understood from JavaScript, Scala, and Kotlin.

```lingo
# Single parameter (no parens needed)
let double = x => x * 2

# Multiple parameters
let add = (a, b) => a + b

# Multi-line lambda with block
let process = x => {
  let validated = validate(x)
  transform(validated)
}

# Type-annotated lambda (rare, for disambiguation)
let parse = (s: Str) => to_int(s)
```

#### 3.4.3 Pipeline Operator

The `|>` operator is a first-class feature. It passes the left-hand value as the first argument
to the right-hand function, replacing nested calls with linear, left-to-right data flow.

**Semantics:** `x |> f(a, b)` desugars to `f(x, a, b)`.

```lingo
# Instead of: sort(filter(map(items, transform), predicate))
# Write:
let result = items
  |> map(transform)
  |> filter(predicate)
  |> sort

# Pipelines with lambdas
let names = users
  |> filter(u => u.age >= 18)
  |> map(u => u.name)
  |> sort
  |> join(", ")

# Pipelines with error handling
let config = read_file(path)
  |> map_err(e => ConfigError::FileNotFound(path))?
```

The standard library is designed with the "primary data argument first" pattern so that all
collection functions compose naturally in pipelines.

#### 3.4.4 Partial Application

Functions support partial application through placeholder syntax:

```lingo
let add_one = add(_, 1)
let double_all = map(_, x => x * 2)

# Equivalent to:
let add_one = x => add(x, 1)
```

#### 3.4.5 Multiple Return Values via Tuples

```lingo
fn divide(a: Float, b: Float) -> Result[(Float, Float), Str] {
  if b == 0.0 { Err("division by zero") }
  else { Ok((a / b, a % b)) }
}

# Destructuring the result
let (quotient, remainder) = divide(10.0, 3.0)?
```

#### 3.4.6 Struct Methods

Methods are defined in `impl` blocks. The `self` parameter receives the instance.

```lingo
struct Point {
  x: Float
  y: Float
}

impl Point {
  fn new(x: Float, y: Float) -> Point {
    Point { x, y }
  }

  fn distance(self, other: Point) -> Float {
    let dx = self.x - other.x
    let dy = self.y - other.y
    sqrt(dx * dx + dy * dy)
  }

  fn translate(self, dx: Float, dy: Float) -> Point {
    Point { x: self.x + dx, y: self.y + dy }
  }
}

# Usage
let p1 = Point::new(0.0, 0.0)
let p2 = Point::new(3.0, 4.0)
let d = p1.distance(p2)    # 5.0
```

### 3.5 Error Handling

Lingo uses `Result` and `Option` types with the `?` operator. There are no exceptions. This
combines minimal happy-path token overhead with explicit fallibility in function signatures.

#### 3.5.1 The `?` Operator

The `?` operator propagates errors from `Result` and `None` from `Option`. It replaces 4 lines
of match-based error handling with 1 character.

```lingo
fn read_config(path: Str) -> Result[Config, Error] {
  let content = read_file(path)?        # propagates file errors
  let parsed = parse_json(content)?     # propagates parse errors
  let config = validate(parsed)?        # propagates validation errors
  Ok(config)
}
```

Without `?`, the equivalent requires explicit match:

```lingo
# This is the verbose form that ? replaces
let content = match read_file(path) {
  Ok(c) => c
  Err(e) => return Err(e)
}
```

The `?` operator can only be used in functions whose return type is `Result` or `Option`. The
compiler enforces this constraint.

#### 3.5.2 Result Combinators

`Result` and `Option` support method-style combinators that compose in pipelines:

```lingo
let value = get_data()
  |> map_err(e => Error::Io(e))     # transform error type
  |> and_then(validate)              # chain fallible operations
  |> unwrap_or(default_value)        # provide fallback
```

**Common combinators:**

| Combinator | Type | Description |
| --- | --- | --- |
| `map` | `Result[T, E] -> (T -> U) -> Result[U, E]` | Transform success value |
| `map_err` | `Result[T, E] -> (E -> F) -> Result[T, F]` | Transform error value |
| `and_then` | `Result[T, E] -> (T -> Result[U, E]) -> Result[U, E]` | Chain fallible operations |
| `unwrap_or` | `Result[T, E] -> T -> T` | Provide fallback on error |
| `ok_or` | `Option[T] -> E -> Result[T, E]` | Convert Option to Result |
| `is_ok` | `Result[T, E] -> Bool` | Check if success |
| `is_some` | `Option[T] -> Bool` | Check if present |

#### 3.5.3 Option Handling

```lingo
fn find_user(name: Str) -> Option[User] {
  users |> find(u => u.name == name)
}

# Using ? with Option (returns None from enclosing function)
fn get_email(name: Str) -> Option[Str] {
  let user = find_user(name)?
  Some(user.email)
}

# Providing defaults
let display_name = find_user(name)
  |> map(u => u.display_name)
  |> unwrap_or("Anonymous")
```

#### 3.5.4 Panic

`panic("message")` terminates the program immediately for unrecoverable errors. It signals
programmer error -- violated invariants, unreachable code -- and is not an error handling
mechanism.

```lingo
fn get_first(items: [Int]) -> Int {
  match items {
    [head, ..] => head
    [] => panic("get_first called on empty list")
  }
}
```

### 3.7 Concurrency Model

Lingo uses structured concurrency with async/await, built on a lightweight task system. This
fits the target domain of CLI tools and data pipelines where concurrent I/O (HTTP requests,
file processing, database queries) is the primary concurrency need.

#### 3.7.1 Async Functions

The `async` keyword marks a function as asynchronous. The `.await` suffix suspends execution
until the future completes.

```lingo
async fn fetch(url: Str) -> Result[Str, Error] {
  let response = http::get(url).await?
  Ok(response.body)
}
```

#### 3.7.2 Task Spawning

`spawn` creates a lightweight concurrent task. `join_all` waits for multiple tasks to complete.

```lingo
async fn fetch_all(urls: [Str]) -> Result[[Str], Error] {
  let tasks = urls |> map(url => spawn(fetch(url)))
  let results = tasks |> join_all.await
  results |> collect_results
}
```

#### 3.7.3 Channels

Channels provide typed, async-safe communication between tasks in producer-consumer patterns.

```lingo
async fn pipeline() {
  let (tx, rx) = channel()

  spawn(async {
    for item in generate_items() {
      tx.send(item).await
    }
  })

  for item in rx {
    process(item).await
  }
}
```

#### 3.7.4 Select

`select` waits on multiple async operations and executes the branch for whichever completes
first.

```lingo
async fn timeout_fetch(url: Str) -> Result[Str, Error] {
  select {
    result = fetch(url) => result
    _ = sleep(5000) => Err(Error::Timeout)
  }
}
```

#### 3.7.5 Structured Concurrency Example

```lingo
async fn process_batch(urls: [Str]) -> Result[[Summary], Error] {
  # Fetch all URLs concurrently, fail fast on first error
  let bodies = urls
    |> map(url => spawn(fetch(url)))
    |> join_all.await
    |> collect_results?

  # Parse results sequentially
  let records = bodies
    |> flat_map(body => body |> split("\n"))
    |> enumerate
    |> map((i, line) => parse_record(line, i))
    |> collect_results?

  Ok(summarize(records))
}
```

### 4.3 Complex: Concurrent Data Pipeline with Error Handling

#### Data Pipeline in Lingo

```lingo
struct Record {
  id: Int
  category: Str
  value: Float
  timestamp: Int
}

struct Summary {
  category: Str
  count: Int
  total: Float
  average: Float
}

enum PipelineError {
  FetchFailed(url: Str, msg: Str)
  ParseFailed(line: Int, msg: Str)
  WriteFailed(path: Str, msg: Str)
}

fn parse_record(line: Str, line_num: Int) -> Result[Record, PipelineError] {
  let parts = line |> split(",")
  match parts |> len {
    4 => {
      let id = parts[0] |> trim |> to_int
        |> ok_or(PipelineError::ParseFailed(line_num, "invalid id"))?
      let category = parts[1] |> trim
      let value = parts[2] |> trim |> to_float
        |> ok_or(PipelineError::ParseFailed(line_num, "invalid value"))?
      let timestamp = parts[3] |> trim |> to_int
        |> ok_or(PipelineError::ParseFailed(line_num, "invalid timestamp"))?
      Ok(Record { id, category, value, timestamp })
    }
    n => Err(PipelineError::ParseFailed(line_num, "expected 4 fields, got {n}"))
  }
}

async fn fetch_data(url: Str) -> Result[Str, PipelineError] {
  http::get(url).await
    |> map_err(e => PipelineError::FetchFailed(url, e.msg))
}

fn summarize(records: [Record]) -> [Summary] {
  records
    |> group_by(r => r.category)
    |> map((cat, recs) => {
      let total = recs |> map(r => r.value) |> fold(0.0, (a, b) => a + b)
      let count = recs |> len
      Summary {
        category: cat
        count: count
        total: total
        average: total / to_float(count)
      }
    })
    |> sort_by(s => s.total)
    |> rev
}

async fn write_report(path: Str, summaries: [Summary]) -> Result[(), PipelineError] {
  let header = "Category,Count,Total,Average"
  let lines = summaries |> map(s =>
    "{s.category},{s.count},{s.total},{s.average}"
  )
  let content = [header] ++ lines |> join("\n")
  write_file(path, content)
    |> map_err(e => PipelineError::WriteFailed(path, e.msg))
}

async fn run_pipeline(urls: [Str], output: Str) -> Result[(), PipelineError] {
  # Fetch all sources concurrently
  let tasks = urls |> map(url => spawn(fetch_data(url)))
  let responses = tasks |> join_all.await |> collect_results?

  # Parse all records
  let records = responses
    |> flat_map(body => body |> split("\n"))
    |> enumerate
    |> map((i, line) => parse_record(line, i + 1))
    |> collect_results?

  # Filter, summarize, and write
  let valid_records = records
    |> filter(r => r.value > 0.0 && r.timestamp > 0)
  let summaries = summarize(valid_records)

  println("Processed {records |> len} records into {summaries |> len} categories")

  write_report(output, summaries).await
}

async fn main() {
  let urls = [
    "https://data.example.com/q1.csv"
    "https://data.example.com/q2.csv"
    "https://data.example.com/q3.csv"
  ]
  match run_pipeline(urls, "report.csv").await {
    Ok(_) => println("Pipeline complete")
    Err(e) => match e {
      PipelineError::FetchFailed(url, msg) =>
        println("Fetch failed ({url}): {msg}")
      PipelineError::ParseFailed(line, msg) =>
        println("Parse error at line {line}: {msg}")
      PipelineError::WriteFailed(path, msg) =>
        println("Write failed ({path}): {msg}")
    }
  }
}
```

**Token count:** ~520 tokens.

## 5. Tradeoff Analysis

Every design decision in Lingo involves a tradeoff between token efficiency, LLM reasoning
quality, and developer experience. This section documents where readability was sacrificed for
savings, where it was preserved, and the empirical justification for each choice.

### 5.1 Where Readability Was Sacrificed for Token Savings

**Short type names (`Str`, `Int`, `Bool` vs. `String`, `Integer`, `Boolean`).**
Both short and long forms are single tokens in cl100k_base, but shorter names produce fewer bytes
in generated output. The sacrifice is minimal: `Str` is universally understood. Research finding:
"For keywords that are not already in tokenizer vocabularies, shorter is better because BPE will
split them into fewer subword tokens."

**No block comments.**
Multi-line comments require repeated `#` prefixes. This is a minor inconvenience for
documentation-heavy code. Justified by structural simplicity: block comment delimiters add
grammar complexity without saving tokens, since comments are typically generated once, not
iterated on.

**`#` instead of `//` for comments.**
Unfamiliar to Rust and JavaScript developers. Both `#` and `//` are single tokens in cl100k_base,
so no token difference. `#` was chosen because it is the Python convention, and Python is the
language LLMs know best (90-97% of LLM-generated code is Python). This tips the familiarity
balance toward the strongest LLM language.

**Mandatory type annotations at function boundaries.**
This costs tokens relative to Python's fully-dynamic approach. Justified by the iteration tax
research: types reduce total cost across the full generate-debug cycle. The type checker catches
errors before they become multi-turn debugging sessions.

### 5.2 Where Readability Was Preserved Despite Token Cost

**Descriptive identifier names.**
Shortening identifiers produces a 41% token savings but causes approximately 50% reasoning
performance collapse. Research shows descriptive names achieve 34.2% exact match rate vs. 16.6%
for minimal names. The unfavorable exchange rate (41% savings for ~50% reasoning loss) makes
identifier compression the single worst available tradeoff. Lingo's conventions explicitly
require descriptive names at module and function boundaries.

**No operator overloading.**
J achieves 70 tokens partly through context-dependent operator meanings. Lingo rejects this:
each operator has exactly one meaning regardless of context. This costs tokens relative to J but
eliminates the semantic ambiguity that degrades LLM reasoning. Research finding: "the degradation
attributed to terseness is actually caused by semantic opacity from overloaded symbols."

**No significant whitespace.**
Python's indentation-as-syntax costs only 6.5% formatting overhead, less than braces. However,
LLMs must count indentation to understand structure, which is error-prone. Lingo uses braces
despite the higher formatting cost because structural clarity for LLMs outweighs the token
difference.

**No implicit topic variables.**
Perl's `$_` and similar implicit state variables save tokens but create invisible data flow that
LLMs cannot trace. Every data flow in Lingo is explicit and traceable.

**No deep point-free composition.**
Haskell-style point-free chains of 5+ composed functions eliminate all naming, destroying
reasoning anchors. Lingo's pipeline operator preserves left-to-right readability and
intermediate naming opportunities.

### 5.3 Quantified Tradeoff Summary

| Decision | Token Cost | Token Savings / Benefit | Net Assessment |
| --- | --- | --- | --- |
| Hindley-Milner inference | Compiler complexity | ~15-20% vs. Java/Go | Large win |
| Pipeline operator | 2 chars per use | 3-8 tokens per chain | Large win |
| `?` error propagation | 1 char per use | 4 lines per call site | Large win |
| Sum types vs. class hierarchies | ~Same as classes | 60-100% fewer tokens vs. Java | Large win |
| Braces vs. indentation | ~5% formatting cost | Structural clarity for LLMs | Small cost |
| Function boundary annotations | ~2-4 tokens/function | Iteration tax reduction | Medium win |
| Descriptive names (not shortened) | 41% more tokens | 2x reasoning quality | Net win |

## 6. PEG Grammar

The following PEG grammar defines Lingo's complete core syntax. Standard PEG notation is used:
`/` for ordered choice, `*` for zero-or-more, `+` for one-or-more, `?` for optional, `&` for
positive lookahead, `!` for negative lookahead.

This grammar is approximately 200 lines of PEG notation, compact enough to include in an LLM
prompt alongside a task description for grammar-prompted generation.

```peg
# =============================================================================
# Lingo PEG Grammar -- Version 0.1.0
# =============================================================================

# -----------------------------------------------------------------------------
# Top Level
# -----------------------------------------------------------------------------

# A program is a sequence of top-level items separated by whitespace.
Program         <- Spacing Item* EOF

# Top-level items: declarations and statements.
Item            <- UseDecl
                 / PubDecl
                 / FnDecl
                 / AsyncFnDecl
                 / StructDecl
                 / EnumDecl
                 / TraitDecl
                 / ImplDecl
                 / ModDecl
                 / TypeAlias
                 / LetStmt

# The pub keyword can precede any declaration to make it public.
PubDecl         <- PUB (FnDecl / AsyncFnDecl / StructDecl / EnumDecl
                        / TraitDecl / TypeAlias)

# -----------------------------------------------------------------------------
# Use / Import
# -----------------------------------------------------------------------------

UseDecl         <- USE UsePath Terminator

# Import paths: grouped {A, B}, glob *, or single item with optional alias.
UsePath         <- ModPath DCOLON LBRACE IdentList RBRACE
                 / ModPath DCOLON STAR
                 / ModPath (AS Ident)?

ModPath         <- Ident (DCOLON Ident)*

IdentList       <- Ident (COMMA Ident)* COMMA?

# -----------------------------------------------------------------------------
# Module
# -----------------------------------------------------------------------------

ModDecl         <- MOD Ident LBRACE Item* RBRACE

# -----------------------------------------------------------------------------
# Type Alias
# -----------------------------------------------------------------------------

TypeAlias       <- TYPE Ident GenericParams? ASSIGN TypeExpr Terminator

# -----------------------------------------------------------------------------
# Functions
# -----------------------------------------------------------------------------

# Synchronous function declaration.
FnDecl          <- FN Ident GenericParams? LPAREN ParamList? RPAREN
                   ReturnType? Block

# Async function declaration.
AsyncFnDecl     <- ASYNC FN Ident GenericParams? LPAREN ParamList? RPAREN
                   ReturnType? Block

ParamList       <- Param (COMMA Param)* COMMA?

# Parameters: either self (for methods) or name: Type.
Param           <- SELF
                 / Ident COLON TypeExpr

# Return type annotation: -> Type
ReturnType      <- ARROW TypeExpr

# -----------------------------------------------------------------------------
# Struct
# -----------------------------------------------------------------------------

StructDecl      <- STRUCT Ident GenericParams? LBRACE FieldList? RBRACE

FieldList       <- Field (Terminator Field)* Terminator?

# Each field has a name and explicit type.
Field           <- Ident COLON TypeExpr

# -----------------------------------------------------------------------------
# Enum (Sum Types)
# -----------------------------------------------------------------------------

EnumDecl        <- ENUM Ident GenericParams? LBRACE VariantList RBRACE

VariantList     <- Variant (Terminator Variant)* Terminator?

# Variants can be bare (Point) or carry data (Circle(radius: Float)).
Variant         <- Ident (LPAREN VariantFields RPAREN)?

VariantFields   <- VariantField (COMMA VariantField)* COMMA?

# Variant fields can be named (radius: Float) or positional (Float).
VariantField    <- (Ident COLON)? TypeExpr

# -----------------------------------------------------------------------------
# Trait
# -----------------------------------------------------------------------------

TraitDecl       <- TRAIT Ident GenericParams? LBRACE TraitItem* RBRACE

# Trait items: full function definitions (defaults) or signatures only.
TraitItem       <- FnDecl / FnSignature

FnSignature     <- FN Ident GenericParams? LPAREN ParamList? RPAREN
                   ReturnType? Terminator

# -----------------------------------------------------------------------------
# Impl
# -----------------------------------------------------------------------------

# impl Type { ... } or impl Trait for Type { ... }
ImplDecl        <- IMPL TypeExpr (FOR TypeExpr)? LBRACE ImplItem* RBRACE

ImplItem        <- FnDecl / AsyncFnDecl

# -----------------------------------------------------------------------------
# Generics
# -----------------------------------------------------------------------------

# Generic parameters use square brackets: fn first[T](items: [T])
GenericParams   <- LBRACK GenericParamList RBRACK

GenericParamList <- GenericParam (COMMA GenericParam)* COMMA?

# Generic param with optional trait bound: T: Display + Numeric
GenericParam    <- Ident (COLON TraitBound)?

TraitBound      <- TypeExpr (PLUS TypeExpr)*

# -----------------------------------------------------------------------------
# Type Expressions
# -----------------------------------------------------------------------------

TypeExpr        <- FnType

# Function types: (A, B) -> C
FnType          <- LPAREN TypeList? RPAREN ARROW TypeExpr
                 / BaseType

TypeList        <- TypeExpr (COMMA TypeExpr)* COMMA?

# Base types: tuples, unit, lists, maps, sets, named types with generics.
BaseType        <- LPAREN TypeExpr (COMMA TypeExpr)+ RPAREN    # Tuple type
                 / LPAREN RPAREN                                # Unit type
                 / LBRACK TypeExpr RBRACK                       # List type
                 / LBRACE TypeExpr COLON TypeExpr RBRACE        # Map type
                 / LBRACE TypeExpr RBRACE                       # Set type
                 / TypePath GenericArgs?

TypePath        <- Ident (DCOLON Ident)*

GenericArgs     <- LBRACK TypeList RBRACK

# -----------------------------------------------------------------------------
# Statements
# -----------------------------------------------------------------------------

Statement       <- LetStmt
                 / ForStmt
                 / WhileStmt
                 / LoopStmt
                 / ExprStmt

# let bindings: let x = expr, let mut x = expr, let x: Type = expr
LetStmt         <- LET MUT? Pattern (COLON TypeExpr)? ASSIGN Expr Terminator

ForStmt         <- FOR Pattern IN Expr Block

WhileStmt       <- WHILE Expr Block

LoopStmt        <- LOOP Block

ExprStmt        <- Expr Terminator

# Statements end with newline, semicolon, or implicitly before }.
Terminator      <- SEMI / NEWLINE / &RBRACE

# -----------------------------------------------------------------------------
# Expressions (precedence climbing)
# -----------------------------------------------------------------------------

# From lowest to highest precedence.
Expr            <- ReturnExpr
                 / BreakExpr
                 / Assignment

ReturnExpr      <- RETURN Expr?
BreakExpr       <- BREAK Expr?

Assignment      <- Pipeline (AssignOp Pipeline)?

AssignOp        <- ASSIGN / PLUS_ASSIGN / MINUS_ASSIGN

# Pipeline: x |> f |> g (left-to-right composition)
Pipeline        <- LogicalOr (PIPE Pipeline)?

LogicalOr       <- LogicalAnd (OR LogicalAnd)*

LogicalAnd      <- Equality (AND Equality)*

Equality        <- Comparison (EqualOp Comparison)?

EqualOp         <- EQEQ / NEQ

Comparison      <- Addition (CompOp Addition)?

CompOp          <- LT / GT / LTEQ / GTEQ

Addition        <- Multiplication (AddOp Multiplication)*

AddOp           <- PLUS / MINUS

Multiplication  <- Unary (MulOp Unary)*

MulOp           <- STAR / SLASH / PERCENT

Unary           <- NOT Unary
                 / MINUS Unary
                 / Postfix

# Postfix operations: ?, .field, .method(), call(), index[], .await
Postfix         <- Primary PostfixOp*

PostfixOp       <- QUESTION                                     # ? operator
                 / DOT Ident (LPAREN ArgList? RPAREN)?          # Method call
                 / DOT Ident                                     # Field access
                 / LPAREN ArgList? RPAREN                        # Function call
                 / LBRACK Expr RBRACK                            # Index
                 / DOT AWAIT                                     # .await

# -----------------------------------------------------------------------------
# Primary Expressions
# -----------------------------------------------------------------------------

Primary         <- IfExpr
                 / MatchExpr
                 / SelectExpr
                 / Block
                 / Lambda
                 / ListLiteral
                 / TupleLiteral
                 / StructLiteral
                 / SpawnExpr
                 / Literal
                 / PathExpr
                 / LPAREN Expr RPAREN

# If/else as expression: if cond { a } else { b }
IfExpr          <- IF Expr Block (ELSE IfExpr / ELSE Block)?

# Pattern matching: match expr { pattern => expr, ... }
MatchExpr       <- MATCH Expr LBRACE MatchArm* RBRACE

MatchArm        <- Pattern Guard? FAT_ARROW Expr Terminator

Guard           <- IF Expr

# Select for async: select { pat = expr => expr, ... }
SelectExpr      <- SELECT LBRACE SelectArm* RBRACE

SelectArm       <- Pattern ASSIGN Expr FAT_ARROW Expr Terminator

# Lambdas: x => expr, (a, b) => { ... }
Lambda          <- LambdaParams FAT_ARROW Expr
                 / LambdaParams FAT_ARROW Block

LambdaParams    <- Ident
                 / LPAREN ParamNameList? RPAREN

ParamNameList   <- Ident (COMMA Ident)* COMMA?

# Blocks: { stmt*; expr? }. The last expression is the block's value.
Block           <- LBRACE Statement* Expr? RBRACE

# List literals: [1, 2, 3] (commas or newlines as separators)
ListLiteral     <- LBRACK (Expr (Terminator Expr)* Terminator?)? RBRACK

# Tuple literals: (a, b) -- trailing comma required for single-element
TupleLiteral    <- LPAREN Expr COMMA (Expr COMMA?)* RPAREN

# Struct literals: TypeName { field: value, ... }
StructLiteral   <- TypePath LBRACE StructFieldInit*
                   (Terminator StructFieldInit)* Terminator? RBRACE

# Field init: name: expr, or just name (shorthand when var matches field)
StructFieldInit <- Ident COLON Expr
                 / Ident

SpawnExpr       <- SPAWN LPAREN Expr RPAREN

ArgList         <- Arg (COMMA Arg)* COMMA?

# Arguments: positional or named, with _ placeholder for partial application
Arg             <- (Ident COLON)? Expr
                 / UNDERSCORE

PathExpr        <- Ident (DCOLON Ident)*

# -----------------------------------------------------------------------------
# Patterns
# -----------------------------------------------------------------------------

Pattern         <- OrPattern

# Or-patterns: Circle(_) | Point => ...
OrPattern       <- BasePattern (PIPE_SYM BasePattern)*

BasePattern     <- LiteralPattern
                 / ConstructorPattern
                 / TuplePattern
                 / ListPattern
                 / StructPattern
                 / RangePattern
                 / BindingPattern
                 / WildcardPattern

LiteralPattern  <- IntLiteral / FloatLiteral / StringLiteral / TRUE / FALSE

# Enum variant destructuring: Some(x), Err(msg)
ConstructorPattern <- TypePath LPAREN PatternList? RPAREN

TuplePattern    <- LPAREN Pattern COMMA (Pattern COMMA?)* RPAREN

# List patterns with optional rest: [first, ..rest], [x, y], []
ListPattern     <- LBRACK ListPatElems? RBRACK

ListPatElems    <- Pattern (COMMA Pattern)* (COMMA DOTDOT Ident?)?

# Struct destructuring: User { name, age, .. }
StructPattern   <- TypePath LBRACE FieldPatterns? RBRACE

FieldPatterns   <- FieldPattern (COMMA FieldPattern)* (COMMA DOTDOT)?

FieldPattern    <- Ident (COLON Pattern)?

# Range patterns in match arms: 1..=10
RangePattern    <- IntLiteral DOTDOT_EQ IntLiteral

BindingPattern  <- Ident

WildcardPattern <- UNDERSCORE

PatternList     <- Pattern (COMMA Pattern)* COMMA?

# -----------------------------------------------------------------------------
# Literals
# -----------------------------------------------------------------------------

Literal         <- FloatLiteral
                 / IntLiteral
                 / StringLiteral
                 / CharLiteral
                 / TRUE
                 / FALSE

IntLiteral      <- [0-9]+

FloatLiteral    <- [0-9]+ '.' [0-9]+

StringLiteral   <- '"' StringChar* '"'
                 / 'r"' RawStringChar* '"'

# String characters: escapes, interpolation {expr}, or plain characters.
StringChar      <- '\\' .                   # Escape sequence
                 / '{' Expr '}'             # Interpolation
                 / !'"' .

RawStringChar   <- !'"' .

CharLiteral     <- "'" . "'"

# -----------------------------------------------------------------------------
# Lexical Elements -- Keywords
# -----------------------------------------------------------------------------

FN              <- 'fn'        !IdentCont Spacing
LET             <- 'let'       !IdentCont Spacing
MUT             <- 'mut'       !IdentCont Spacing
IF              <- 'if'        !IdentCont Spacing
ELSE            <- 'else'      !IdentCont Spacing
MATCH           <- 'match'     !IdentCont Spacing
FOR             <- 'for'       !IdentCont Spacing
IN              <- 'in'        !IdentCont Spacing
WHILE           <- 'while'     !IdentCont Spacing
LOOP            <- 'loop'      !IdentCont Spacing
BREAK           <- 'break'     !IdentCont Spacing
RETURN          <- 'return'    !IdentCont Spacing
TYPE            <- 'type'      !IdentCont Spacing
STRUCT          <- 'struct'    !IdentCont Spacing
ENUM            <- 'enum'      !IdentCont Spacing
TRAIT           <- 'trait'     !IdentCont Spacing
IMPL            <- 'impl'     !IdentCont Spacing
MOD             <- 'mod'       !IdentCont Spacing
USE             <- 'use'       !IdentCont Spacing
PUB             <- 'pub'       !IdentCont Spacing
ASYNC           <- 'async'     !IdentCont Spacing
AWAIT           <- 'await'     !IdentCont Spacing
SELF            <- 'self'      !IdentCont Spacing
AS              <- 'as'        !IdentCont Spacing
SELECT          <- 'select'    !IdentCont Spacing
SPAWN           <- 'spawn'     !IdentCont Spacing
TRUE            <- 'true'      !IdentCont Spacing
FALSE           <- 'false'     !IdentCont Spacing

# -----------------------------------------------------------------------------
# Lexical Elements -- Operators and Punctuation
# -----------------------------------------------------------------------------

# Multi-character operators must be matched before their single-char prefixes.
PIPE            <- '|>'   Spacing
FAT_ARROW       <- '=>'   Spacing
ARROW           <- '->'   Spacing
DCOLON          <- '::'   Spacing
DOTDOT_EQ       <- '..='  Spacing
DOTDOT          <- '..'   Spacing
DOT             <- '.'    Spacing
EQEQ            <- '=='   Spacing
NEQ             <- '!='   Spacing
LTEQ            <- '<='   Spacing
GTEQ            <- '>='   Spacing
LT              <- '<'    !['='] Spacing
GT              <- '>'    !['='] Spacing
AND             <- '&&'   Spacing
OR              <- '||'   Spacing
PLUS_ASSIGN     <- '+='   Spacing
MINUS_ASSIGN    <- '-='   Spacing
PLUS            <- '+'    !['='] Spacing
MINUS           <- '-'    !['='>] Spacing
STAR            <- '*'    Spacing
SLASH           <- '/'    Spacing
PERCENT         <- '%'    Spacing
NOT             <- '!'    !['='] Spacing
ASSIGN          <- '='    !['='>] Spacing
QUESTION        <- '?'    Spacing
COLON           <- ':'    ![':'] Spacing
PIPE_SYM        <- '|'    !['>''|'] Spacing
COMMA           <- ','    Spacing
SEMI            <- ';'    Spacing
LBRACE          <- '{'    Spacing
RBRACE          <- '}'    Spacing
LPAREN          <- '('    Spacing
RPAREN          <- ')'    Spacing
LBRACK          <- '['    Spacing
RBRACK          <- ']'    Spacing
UNDERSCORE      <- '_'    !IdentCont Spacing
HASH            <- '#'

# -----------------------------------------------------------------------------
# Lexical Elements -- Identifiers
# -----------------------------------------------------------------------------

# Identifiers cannot be keywords.
Ident           <- !Keyword IdentStart IdentCont* Spacing

IdentStart      <- [a-zA-Z_]
IdentCont       <- [a-zA-Z0-9_]

Keyword         <- ('fn' / 'let' / 'mut' / 'if' / 'else' / 'match' / 'for'
                 / 'in' / 'while' / 'loop' / 'break' / 'return' / 'type'
                 / 'struct' / 'enum' / 'trait' / 'impl' / 'mod' / 'use'
                 / 'pub' / 'async' / 'await' / 'self' / 'as' / 'select'
                 / 'spawn' / 'true' / 'false') !IdentCont

# -----------------------------------------------------------------------------
# Lexical Elements -- Whitespace and Comments
# -----------------------------------------------------------------------------

Spacing         <- (Whitespace / Comment)*
Whitespace      <- [ \t\r] / NEWLINE
NEWLINE         <- '\n'
Comment         <- '#' (!'\n' .)* '\n'?

EOF             <- !.
```

### Grammar Notes

1. **Terminator handling.** Statements are terminated by newlines, semicolons, or an implicit
   terminator before a closing brace. This allows both single-line and multi-line styles without
   requiring explicit separators.

2. **Expression vs. statement ambiguity.** The last element in a block is treated as an
   expression (the implicit return value). All other elements are statements requiring
   terminators.

3. **Pipeline precedence.** The pipeline operator `|>` has lower precedence than logical
   operators but higher than assignment. `x |> f |> g` chains naturally; `let y = x |> f` binds
   correctly.

4. **String interpolation.** Interpolation is handled at the lexical level within `StringChar`.
   The `{Expr}` production inside strings allows arbitrary expressions, making the grammar
   context-sensitive at that point. A parser handles this by switching to expression parsing mode
   upon encountering `{` within a string.

5. **Newline significance.** Newlines serve as statement terminators (like Go) but are not
   significant for structure (unlike Python). Consecutive newlines collapse. A newline is
   suppressed as a terminator after continuation tokens: `|>`, `=>`, binary operators, `(`,
   `[`, `{`, `,`.

6. **Grammar size.** This grammar is approximately 200 lines of PEG notation, compact enough
   to include in an LLM prompt alongside a task description. This supports the grammar-prompted
   generation cold-start strategy.

## 7. Quick Reference

### Declarations

```lingo
# Variable binding (immutable)
let x = 42
let name = "Alice"

# Mutable binding
let mut count = 0
count += 1

# Function
fn add(a: Int, b: Int) -> Int { a + b }

# Async function
async fn fetch(url: Str) -> Result[Str, Error] { ... }

# Struct
struct Point { x: Float, y: Float }

# Enum (sum type)
enum Shape {
  Circle(radius: Float)
  Rect(width: Float, height: Float)
  Point
}

# Trait
trait Display { fn display(self) -> Str }

# Implementation
impl Display for Point {
  fn display(self) -> Str { "({self.x}, {self.y})" }
}

# Type alias
type Id = Int
```

### Types

```text
Int           64-bit integer
Float         64-bit float
Bool          true / false
Str           UTF-8 string
Char          Unicode scalar
[T]           List
{K: V}        Map
{T}           Set
(A, B)        Tuple
()            Unit
Option[T]     Some(T) | None
Result[T, E]  Ok(T) | Err(E)
```

### Control Flow

```lingo
# If/else (expression)
let x = if cond { a } else { b }

# Match (exhaustive)
match value {
  Pattern => expr
  Pattern if guard => expr
  _ => default
}

# Loops
for item in collection { ... }
for i in 0..10 { ... }
while condition { ... }
loop { ... }
```

### Functions and Lambdas

```lingo
# Named function
fn name(param: Type) -> ReturnType { body }

# Lambda
x => x + 1
(a, b) => a + b

# Pipeline
data |> transform |> filter(pred) |> sort

# Partial application
let f = add(_, 1)
```

### Error Handling

```lingo
# Propagate with ?
let value = fallible_call()?

# Match on result
match result {
  Ok(v) => use(v)
  Err(e) => handle(e)
}

# Combinators
value |> map_err(transform) |> unwrap_or(default)
```

### Concurrency

```lingo
# Async function
async fn work() -> Result[T, E] { ... }

let result = async_call().await

let task = spawn(async_work())

let (tx, rx) = channel()
tx.send(value).await
for msg in rx { ... }

select {
  result = fetch(url) => result
  _ = sleep(5000) => Err(Error::Timeout)
}
```

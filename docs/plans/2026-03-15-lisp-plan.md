# Plan: Lingo to Lisp Transformation (2026-03-15)

## Summary

Transform Lingo from its current imperative/functional syntax (curly braces,
infix operators, Go-style semicolons) into a Lisp with S-expression syntax
and evaluate/apply semantics. This is a near-total rewrite of the core
pipeline (lexer, AST, parser, interpreter) followed by updates to supporting
infrastructure (REPL, test runner, main.rs) and a full rewrite of all tests
and examples. The resulting codebase should shrink from approximately 3,175
lines of Rust to approximately 1,200 lines -- a 62% reduction -- while
gaining homoiconicity, first-class operators, and macro-readiness.

## Stakes Classification

**Level**: High

**Rationale**: This change replaces almost the entire codebase. Every module
except `lib.rs` changes significantly. The lexer, AST, parser, and
interpreter are complete rewrites. All 50 integration tests and 4 example
programs must be rewritten. There is no incremental migration path -- the
old syntax and new syntax are fundamentally incompatible. However, risk is
mitigated by the fact that the new code is dramatically simpler than the old,
and every phase produces a compilable, testable state.

## Key Design Decisions

These resolve the open questions from the research document:

1. **Naming convention**: Use hyphens for builtin names (Lisp convention):
   `assert-eq`, `read-line`, `sort-by`, etc. Update test runner discovery
   to match `test-` prefix with hyphens.

2. **Boolean literals**: Use `true`/`false` (not `#t`/`#f`). Simpler,
   matches existing `Value::Bool`, avoids introducing Scheme conventions
   that serve no purpose.

3. **File extension**: Keep `.ln`. No reason to change -- the file extension
   is arbitrary and changing it creates unnecessary churn.

4. **Tail call optimization**: Defer. The current interpreter uses Rust's
   call stack with no TCO, and this works fine for the use cases. TCO via
   trampolining can be added later.

5. **Macro system**: Defer. The architecture is macro-ready (homoiconic
   AST), but macros add significant complexity and are not needed for the
   initial transformation.

6. **Rc/RefCell env optimization**: Defer. The current clone-based
   `Box<Env>` approach works for a small interpreter. Optimize later if
   profiling shows it matters.

7. **Special forms**: Include `while` and a simple `match` (literal +
   wildcard patterns) as special forms from the start. The full set is:
   `define`, `defn`, `fn`, `if`, `cond`, `let`, `begin`, `quote`, `set!`,
   `and`, `or`, `while`, `match`, `->` (threading).

8. **Threading macro**: Include `->` from the start. It maps to Lingo's
   pipeline operator (`|>`) and preserves the linear-composition design
   goal that was central to Lingo.

## Implementation Steps

### Phase 1: Core Pipeline Rewrite (lexer, AST, parser, interpreter)

This is the largest phase. It replaces the four core modules with their Lisp
equivalents. The order is bottom-up: AST first (no dependencies), then lexer
(depends on nothing), then parser (depends on AST + lexer), then interpreter
(depends on AST).

Each step within this phase builds on the previous one, so they must be
sequential. The phase ends with a compilable project where the core pipeline
works but supporting infrastructure (REPL, test runner, main.rs) is not yet
updated.

#### Step 1.1: Rewrite `src/ast.rs` -- the Expr enum

- **Files**: `src/ast.rs`
- **Action**: Replace the entire file. The current 153-line file with 30+
  types (`Program`, `Item`, `FnDecl`, `Param`, `Block`, `Stmt`, `Expr`,
  `MatchArm`, `StringPart`, `Literal`, `BinOp`, `UnaryOp`, `Pattern`,
  `Span`) collapses to a single `Expr` enum with 7 variants: `Int(i64)`,
  `Float(f64)`, `Str(String)`, `Bool(bool)`, `Symbol(String)`,
  `List(Vec<Expr>)`, and `Nil`.
  Add helper methods: `as_symbol() -> Result<&str, String>` and
  `as_list() -> Result<&[Expr], String>` for use by special form handlers.
  Derive `Debug` and `Clone`. Implement `Display` for readable output
  of S-expressions.
- **Test cases**: No standalone tests -- this is a data type. Verified by
  compilation in subsequent steps.
- **Verify**: `cargo check` passes (with expected errors in downstream
  modules that still reference old types).
- **Complexity**: Small

#### Step 1.2: Rewrite `src/lexer.rs` -- S-expression tokenizer

- **Files**: `src/lexer.rs`
- **Action**: Replace the entire file. The current 623-line lexer with 86
  token variants becomes approximately 100 lines with 6 token types:
  `LParen`, `RParen`, `Atom(String)`, `Str(String)`, `Quote`, `Eof`.

  The tokenization algorithm is a single character-dispatch loop:
  - Whitespace and commas: skip (commas are optional whitespace)
  - `;`: line comment (skip to end of line)
  - `(` / `)`: emit delimiter token
  - `'`: emit Quote token
  - `"`: consume string with escape sequences (`\n`, `\t`, `\\`, `\"`)
  - Anything else: consume while not whitespace/paren/quote/comma/semicolon,
    emit as `Atom`

  The `Lexer` struct keeps `source: Vec<char>`, `pos`, `line`, `col` with
  `peek`/`advance` methods (same structural approach as current, but
  stripped down). Remove `insert_newlines()`, all multi-char operator
  recognition, string interpolation, and the keyword table.

  Public API: `Lexer::new(source) -> Lexer` and
  `Lexer::tokenize() -> Result<Vec<Token>, String>`.
- **Test cases**: No standalone tests at this step -- verified through
  parser tests in Step 1.3 and interpreter tests in Step 1.5.
- **Verify**: `cargo check` passes (with expected errors in parser and
  interpreter modules).
- **Complexity**: Small

#### Step 1.3: Rewrite `src/parser.rs` -- S-expression reader

- **Files**: `src/parser.rs`
- **Action**: Replace the entire file. The current 901-line recursive
  descent parser with 13 precedence levels becomes approximately 80 lines
  with three functions:

  - `parse(tokens) -> Result<Vec<Expr>, String>`: Read all top-level
    expressions until EOF.
  - `read_expr(tokens, pos) -> Result<Expr, String>`: Dispatch on current
    token: `LParen` reads list elements until `RParen`; `Quote` reads
    next expr and wraps in `(quote ...)`; `Atom` parses as number (i64
    then f64), boolean (`true`/`false`), `nil`, or symbol; `Str` wraps
    in `Expr::Str`; `RParen` errors (unexpected close paren); `Eof`
    errors (unexpected end of input).
  - `read_list(tokens, pos) -> Result<Vec<Expr>, String>`: Read
    expressions until `)` is encountered.

  Public API: `Parser::new(tokens) -> Parser` and
  `Parser::parse() -> Result<Vec<Expr>, String>`.

  Note: The parser no longer returns a `Program` struct. It returns
  `Vec<Expr>` -- a flat list of top-level S-expressions.
- **Test cases**: No standalone parser tests at this step -- verified
  through interpreter integration tests in Phase 3.
- **Verify**: `cargo check` passes (with expected errors in interpreter,
  REPL, main, and test modules that use the old API).
- **Complexity**: Small

#### Step 1.4: Rewrite `src/interpreter.rs` -- evaluate/apply model

This is the largest single step. It replaces the 1,461-line interpreter
with approximately 800 lines organized around the evaluate/apply model.

- **Files**: `src/interpreter.rs`
- **Action**: Replace the entire file. The rewrite has four logical
  sections:

  **Section A: Value enum and Env** (~100 lines)

  The `Value` enum simplifies from 10 variants to 8: `Int(i64)`,
  `Float(f64)`, `Str(String)`, `Bool(bool)`, `Nil`, `List(Vec<Value>)`,
  `Lambda { params: Vec<String>, body: Vec<Expr>, closure: Env }`, and
  `Builtin { name: String, func: fn(&[Value]) -> Result<Value, String> }`.

  Key changes:
  - `Unit` becomes `Nil`
  - `Tuple` is removed (use `List`)
  - `Fn` and `Lambda` merge into a single `Lambda` variant. Named functions
    are lambdas bound via `define`. Params become `Vec<String>` (no more
    `Param` struct with type annotations). Body becomes `Vec<Expr>` to
    support multi-expression bodies (implicit `begin`).
  - `BuiltinFn(String)` becomes `Builtin { name, func }` with an actual
    function pointer.

  Implement `Display`, `Debug`, `Clone` for `Value`. For `Debug` on
  `Builtin`, use the `name` field. For `PartialEq`, compare by name for
  builtins.

  The `Env` struct keeps its current design: `HashMap<String, Value>`
  bindings with `Option<Box<Env>>` parent. Methods: `new()`, `child()`,
  `set()`, `get()`, `update()`, `binding_names()`. No changes to the
  environment model.

  Remove the `Signal` enum (`Return`, `Break`). In a Lisp, functions
  return the last expression; `while` uses a loop internally and returns
  `Nil`. There is no `return` or `break` statement.

  **Section B: Core evaluator** (~80 lines)

  A single `evaluate(expr, env) -> Result<Value, String>` function:
  - `Expr::Int/Float/Str/Bool/Nil`: self-evaluating, return as `Value`
  - `Expr::Symbol(name)`: look up in env, error if unbound
  - `Expr::List(elems)` where empty: return `Nil`
  - `Expr::List(elems)`: check head for special form name, else
    evaluate all elements and apply function

  An `apply_function(func, args) -> Result<Value, String>` function:
  - `Value::Lambda { params, body, closure }`: create child env of
    closure, bind params to args, evaluate body expressions in sequence,
    return last result
  - `Value::Builtin { func, .. }`: invoke `func(args)`
  - Otherwise: error "not a function"

  **Section C: Special form handlers** (~200 lines)

  14 special forms, each a function
  `eval_<form>(args, env) -> Result<Value, String>`:

  1. `define`: `(define name expr)` -- evaluate expr, bind in env
  2. `defn`: `(defn name (params) body...)` -- sugar for define + fn
  3. `fn`: `(fn (params) body...)` -- create Lambda closure
  4. `if`: `(if test then [else])` -- evaluate test, branch
  5. `cond`: `(cond (test1 expr1) ... (else exprN))` -- multi-branch
  6. `let`: `(let ((x 1) (y 2)) body...)` -- local bindings
  7. `begin`: `(begin e1 e2 ... en)` -- sequence, return last
  8. `quote`: `(quote expr)` -- return expr as data (convert to Value)
  9. `set!`: `(set! name expr)` -- mutate existing binding
  10. `and`: `(and a b c)` -- short-circuit, return last truthy or first
      falsy
  11. `or`: `(or a b c)` -- short-circuit, return first truthy or last
      falsy
  12. `while`: `(while test body...)` -- loop while test is truthy,
      return Nil
  13. `match`: `(match expr (pattern1 result1) ... (_ default))` --
      literal + wildcard pattern matching
  14. `->`: `(-> val (f a b) (g c))` desugars to `(g (f val a b) c)` --
      threading macro

  For `quote`, implement `expr_to_value(expr) -> Value` that converts
  an `Expr` tree to a `Value` tree (symbols become strings in a list
  structure, or introduce a quoted symbol representation).

  For `match`, compare scrutinee value against literal patterns
  (Int, Float, Str, Bool) and bind wildcard `_`. No constructor patterns,
  no list patterns, no guards.

  **Section D: Builtin functions** (~400 lines)

  Each builtin is a standalone function with signature
  `fn(&[Value]) -> Result<Value, String>`. A `register_builtins(env)`
  function populates the environment.

  Builtins organized by category:

  *Arithmetic (8)*: `+` (variadic), `-` (variadic: unary negate or
  subtract), `*` (variadic), `/`, `mod`, `abs`, `min`, `max`

  *Comparison (5)*: `=` (structural equality), `<`, `>`, `<=`, `>=`

  *Logic (1)*: `not`

  *List (22)*: `list`, `cons`, `first`, `rest`, `nth`, `length`,
  `append`, `reverse`, `map`, `filter`, `fold`, `for-each`, `flatten`,
  `zip`, `take`, `drop`, `sort`, `sort-by`, `any?`, `all?`, `find`,
  `unique`, `chunk`, `enumerate`, `group-by`, `range`

  *String (12)*: `str` (concatenation, replaces string interpolation),
  `string-length`, `substring`, `split`, `join`, `trim`, `contains?`,
  `replace`, `starts-with?`, `ends-with?`, `upper-case`, `lower-case`

  *Type (8)*: `type-of`, `int?`, `float?`, `string?`, `bool?`, `list?`,
  `nil?`, `number?`

  *Conversion (3)*: `->int`, `->float`, `->str`

  *I/O (5)*: `println`, `print`, `read-line`, `read-file`, `write-file`

  *Debug/Test (5)*: `dbg`, `assert`, `assert-eq`, `assert-ne`,
  `assert-true`, `assert-false`

  Note: Higher-order builtins (`map`, `filter`, `fold`, `sort-by`,
  `for-each`, `any?`, `all?`, `find`, `group-by`) need access to the
  evaluator to call lambda arguments. This requires either:
  (a) passing a callback/evaluator reference, or
  (b) making these special forms instead of builtins.

  **Recommendation**: Make `map`, `filter`, `fold`, `for-each`, `sort-by`,
  `any?`, `all?`, `find`, `group-by` take `Value::Lambda` or
  `Value::Builtin` arguments and invoke them directly via
  `apply_function`. This means the builtin signatures need access to the
  `Env` or the builtins need to call `apply_function` internally. The
  cleanest approach: keep these as builtins with signature
  `fn(&[Value]) -> Result<Value, String>`, and inside their
  implementation, match the function argument as `Value::Lambda` or
  `Value::Builtin` and call it. The Lambda case creates a child env of
  the closure and evaluates the body. Factor this into a shared
  `call_value(func, args) -> Result<Value, String>` helper that both
  `apply_function` and builtins can use.

  **Public API**: The `Interpreter` struct exposes:
  - `Interpreter::new() -> Self` (creates env, registers builtins)
  - `run(source: &str) -> Result<(), String>` (lex, parse, evaluate all
    top-level forms; call `main` if defined)
  - `run_source(source: &str) -> Result<Vec<Value>, String>` (lex,
    parse, evaluate all forms, return results -- used by test
    infrastructure)
  - `eval_source(source: &str) -> Result<Value, String>` (lex, parse,
    evaluate, return last result -- used by REPL and tests)
  - `load_source(source: &str)` (lex, parse, evaluate all forms but do
    not call main -- used by test runner)
  - `call_function(func: &Value, args: &[Value]) -> Result<Value, String>`
    (used by test runner)
  - Public access to `env` field (pub(crate)) for test runner discovery.

- **Test cases**: No standalone tests at this step. The interpreter is
  verified through integration tests in Phase 3. However, the code must
  compile and the following smoke test must pass manually in the REPL
  (verified after Phase 2): `(+ 1 2)` returns `3`;
  `(define x 10)` then `x` returns `10`;
  `(defn double (x) (* x 2))` then `(double 21)` returns `42`.
- **Verify**: `cargo check` passes. All four core modules
  (`ast.rs`, `lexer.rs`, `parser.rs`, `interpreter.rs`) compile cleanly.
  Downstream modules (`repl.rs`, `test_runner.rs`, `main.rs`,
  `tests/test_runner.rs`) will have compilation errors -- that is expected
  and addressed in Phase 2.
- **Complexity**: Large

#### Step 2.4: Update `src/repl.rs` -- paren-counting multi-line input

- **Files**: `src/repl.rs`
- **Action**: Rewrite the REPL to use the new pipeline. Key changes:

  1. **Pipeline**: Replace `Lexer -> Parser -> eval_item` with
     `interpreter.eval_source(buffer)`.

  2. **Multi-line detection**: Replace the error-message sniffing
     (`is_incomplete_input` checking for "None" or "Eof") with
     **paren counting**. Before attempting to parse, count unmatched
     open parens in the buffer. If count > 0, prompt for continuation.
     This is more reliable than error-message parsing.

     Paren counting function: `fn unmatched_parens(input: &str) -> i32`.
     Walk the string, increment for `(`, decrement for `)`, skip
     characters inside string literals (track `"` with escape handling).
     Return the count. If > 0, input is incomplete. If < 0, there is a
     syntax error (extra close parens), but let the parser report it.

  3. **Result printing**: Print non-Nil results. The current check is
     `!matches!(value, Value::Unit)` -- change to
     `!matches!(value, Value::Nil)`.

  4. **Prompts**: Keep `>>` and `..` (or change to `>` and continuing
     spaces). This is a cosmetic choice.

  The overall REPL structure (read line, accumulate buffer, try
  parse/eval, print result, loop) stays the same.
- **Test cases**: Verified by REPL integration tests in Phase 3.
- **Verify**: `cargo build` succeeds with zero errors. The binary starts
  and the REPL accepts input.
- **Complexity**: Small

#### Step 4.2: Rewrite `examples/basics.ln`

- **Files**: `examples/basics.ln`
- **Action**: Rewrite in Lisp syntax. This demonstrates arithmetic,
  conditionals, lists, higher-order functions, function definitions,
  string operations, and pattern matching.

  Current features to translate:
  - Function with type annotations -> `defn` (drop type annotations)
  - String interpolation -> `str` builtin + `println`
  - `if/else` expression -> `(if ...)`
  - Pipeline `|>` with `map`, `filter`, `fold` -> `->` threading macro

  ```scheme

  (defn main ()
    ; Arithmetic
    (define x 10)
    (define y 3)

    (println (str "x % y = " (->str (mod x y))))

    (define doubled (-> nums (map (fn (n) (* n 2)))))
    (println (str "doubled: " (->str doubled)))

    (define evens (-> nums (filter (fn (n) (= (mod n 2) 0)))))
    (println (str "evens: " (->str evens)))

    (define total (-> nums (fold 0 (fn (acc n) (+ acc n)))))
    (println (str "total: " (->str total)))

    ; Function calls
    (println (str "double(21) = " (->str (double 21))))

    ; String operations
    (define greeting "Hello, World!")
    (println (str "length: " (->str (string-length greeting))))
    (println (str "upper: " (upper-case greeting)))

    ; Pattern matching
    (define items (list 1 2 3))
    (define desc (match (length items)
      (0 "empty")
      (1 "singleton")
      (_ "multiple")))
    (println (str "items: " desc)))
  ```

## Risks and Mitigations

| Risk | Impact | Mitigation |
| --- | --- | --- |
| Higher-order builtins need evaluator access | Blocks builtin impl | Factor `call_value` helper; values carry state |
| Paren counting may miss strings with parens | Multi-line breaks | String-aware paren counter tracking quote state |
| `define` inside function bodies | Local var issues | Works: `defn` bodies run in child env |
| Discovery change `test_` to `test-` | Old tests break | Intentional; all test files rewritten |
| No `return` or `break` statements | Translation limits | Use `if` for conditional returns; restructure loops |
| `quote` semantics for nested structures | Non-trivial conversion | Implement `expr_to_value` mapper |
| Debug/Clone for Value with fn pointers | Derive issues | Manual Debug impl using name field |
| Loss of string interpolation | Verbose syntax | Deliberate; `(str ...)` is clear and explicit |

## Rollback Strategy

This is a full rewrite with no incremental migration path. Rollback options:

1. **Git revert**: The entire transformation should be done on a feature
   branch. If the transformation fails or is abandoned, the branch can be
   deleted and main remains unchanged.

2. **Phase-level rollback**: Each phase produces a compilable state. If a
   later phase encounters blocking issues, earlier phases can be kept and
   the problematic phase reworked.

3. **Pre-transformation snapshot**: Tag the current state before starting
   (`git tag pre-lisp-transformation`) so it can be restored at any time.

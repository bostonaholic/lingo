# Plan: Lingo REPL (2026-03-15)

## Summary

Add a Read-Eval-Print Loop (REPL) to the Lingo programming language. When the
`lingo` binary is invoked with no file argument, it enters an interactive
session that reads input, parses and evaluates it using the existing
lexer/parser/interpreter pipeline, and prints non-`Unit` results. The REPL
persists interpreter state across inputs (variable bindings, function
declarations), recovers gracefully from parse and evaluation errors, supports
multi-line input by accumulating lines when the parser reports unexpected EOF,
and exits cleanly on Ctrl-D/EOF.

## Stakes Classification

**Level**: Medium

**Rationale**: The REPL touches three files -- `src/main.rs` (CLI entry point
change), `src/interpreter.rs` (one new public method), and a new
`src/repl.rs` module -- but each change is small and additive. The existing
`run()` and file-execution paths are untouched. The interpreter's internal
evaluation methods remain private; only a thin public wrapper is added. The
REPL has no effect on existing tests, existing CLI behavior (when a file
argument is provided), or the test framework. Rollback is trivial: delete the
new module and revert two small changes.

## Design Decisions

### Enter REPL when no file argument is given

The current behavior when no file is provided is to print a usage message and
exit with code 1. The REPL replaces this: `lingo` with no arguments starts the
interactive session. The `--test` flag still requires a file argument.

### Prompt format: `>>` and `..`

The primary prompt is `>>` followed by a space. When the REPL is accumulating
multi-line input, the continuation prompt is `..` followed by a space. These
are short, visually distinct, and do not collide with Lingo syntax.

### Multi-line detection via `got None` in parser errors

When the parser runs out of tokens mid-expression (e.g., an unclosed brace or
incomplete function declaration), it produces an error containing `got None`
(from the `expect` method) or `Unexpected token None` (from the primary
parser). The REPL detects these patterns in the error string and switches to
continuation mode, accumulating additional lines until either:

1. Parsing succeeds, or
2. A non-EOF parse error occurs (displayed to the user), or
3. The user submits an empty line (cancels the multi-line input).

This approach avoids modifying the parser. Since errors are plain `String`
values, substring matching on `"None"` is sufficient. The parser error format
is stable within the crate.

### New public method: `Interpreter::eval_item`

The existing `run()` method returns `Result<(), String>` -- it discards
expression values. The REPL needs the value back. Rather than making the
private `eval_expr` public (which would expose too much surface area), a new
method `eval_item(&mut self, item: &Item) -> Result<Value, String>` is added.
It handles both `Item::FnDecl` (registers the function, returns `Unit`) and
`Item::ExprStmt` (evaluates the expression, returns the result). This keeps
the public API minimal and purposeful.

### Print non-Unit values

After evaluating each item, the REPL prints the resulting `Value` using its
`Display` implementation, but only if the value is not `Value::Unit`. This
means `let x = 5` and `println("hi")` produce no extra output, while `1 + 2`
prints `3` and `[1, 2, 3] |> map(n => n * 2)` prints `[2, 4, 6]`.

### No `main()` auto-invocation in REPL

In file mode, if a `main()` function exists, it is called automatically. In
REPL mode, defining `fn main() { ... }` simply registers the function -- the
user must call `main()` explicitly. This is consistent with how a REPL should
behave: declarations are declarations, not executions.

### Module placement: `src/repl.rs`

The REPL logic lives in its own module (`src/repl.rs`), following the existing
one-file-per-module convention (`lexer.rs`, `parser.rs`, `interpreter.rs`,
`test_runner.rs`). The module is declared in `src/lib.rs` and invoked from
`src/main.rs`.

to an existing module's interface and must be done first since the REPL module
depends on it.

- **Files**: `tests/test_runner.rs`
- **Action**: Add tests that create an `Interpreter`, parse a source string
  into a `Program`, and call `eval_item` on each item, verifying the returned
  `Value`.
- **Test cases**:
  - Expression `1 + 2` returns `Value::Int(3)` (via `Display` format `"3"`)
  - Expression `"hello"` returns `Value::Str("hello")`
  - `let x = 5` returns `Value::Unit`
  - `fn foo() { 42 }` returns `Value::Unit` (declaration only)
  - After declaring `fn foo() { 42 }`, calling `foo()` returns `Value::Int(42)`
  - After `let x = 10`, evaluating `x` returns `Value::Int(10)`
  - Evaluating an undefined variable returns `Err`
- **Verify**: Tests compile and fail (method does not exist yet)
- **Complexity**: Small

#### Step 1.2: Implement `eval_item` method (GREEN)

- **Files**: `src/interpreter.rs` (near `run()` at line ~186)
- **Action**: Add a public method to `Interpreter`:

  ```rust
  pub fn eval_item(&mut self, item: &Item) -> Result<Value, String> {
      match item {
          Item::FnDecl(f) => {
              let val = Value::Fn {
                  name: f.name.clone(),
                  params: f.params.clone(),
                  body: f.body.clone(),
                  closure: self.env.clone(),
              };
              self.env.set(f.name.clone(), val);
              Ok(Value::Unit)
          }
          Item::ExprStmt(expr) => self.eval_expr(expr),
      }
  }
  ```

- **Verify**: All tests from Step 1.1 pass
- **Complexity**: Small

### Phase 2: REPL Module and CLI Integration

Create the REPL module and wire it into the CLI entry point.

#### Step 2.1: Test REPL behavior via CLI (RED)

- **Files**: `tests/test_runner.rs`
- **Action**: Write integration tests that invoke the `lingo` binary with no

- **Test cases**:

  - `lingo file.ln` still works (regression -- already covered by existing

- **Verify**: Tests compile and fail (REPL does not exist yet)
- **Complexity**: Medium

#### Step 2.2: Implement REPL module (GREEN)

- **Files**: `src/repl.rs` (new file), `src/lib.rs`
- **Action**: Create `src/repl.rs` with:

  1. `pub fn start() -> Result<(), String>` -- the REPL entry point.
  2. Uses `std::io::{self, BufRead, Write}` for stdin/stdout.
  3. Creates a single `Interpreter::new()` that persists for the session.
  4. Main loop:
     - Print the `>>` prompt, flush stdout.
     - Read a line from stdin. On EOF (`None` from `lines()`), break
       cleanly.
     - If the line is empty (whitespace only), continue to next iteration.
     - Lex the input. On lex error, print the error to stderr and continue.
     - Parse the tokens. If parsing fails with an error containing `"None"`
       (indicating unexpected EOF), enter multi-line mode:
       - Print the `..` continuation prompt.
       - Read additional lines, appending to the buffer (with newlines).
       - Re-lex and re-parse after each new line.
       - Exit multi-line mode when: parse succeeds, a non-EOF error occurs,
         or the user enters an empty line.
     - For each `Item` in the parsed `Program`, call
       `interpreter.eval_item(item)`.
     - If `eval_item` returns `Ok(value)` and value is not `Value::Unit`,
       print the value to stdout.
     - If `eval_item` returns `Err(msg)`, print the error to stderr and
       continue.
  5. Add `pub mod repl;` to `src/lib.rs`.

- **Verify**: All tests from Step 2.1 pass
- **Complexity**: Medium

## Risks and Mitigations

**Multi-line detection via string matching on `"None"` is fragile if
parser error format changes.**
Impact: Low -- parser is within the same crate and error format is stable.
Mitigation: Keep detection logic in one function; if parser errors are ever
structured, update the check in one place.

**`eval_item` exposes `eval_expr` indirectly through a public method.**
Impact: Low -- `eval_item` is a thin wrapper with a well-defined contract.
Mitigation: The method only accepts `Item`, not arbitrary `Expr`; internal
evaluation remains private.

**Stdin line buffering may interact poorly with multi-line blocks.**
Impact: Low -- `BufRead::lines()` handles line buffering correctly.
Mitigation: Test with piped input in integration tests to verify.

**REPL prompt interleaves with `println` output from evaluated code.**
Impact: Low -- both write to stdout, which is expected.
Mitigation: No mitigation needed; this is standard REPL behavior.

**Changing the no-argument behavior from "usage error" to "REPL" could
surprise users expecting the old behavior.**
Impact: Low -- REPLs are the conventional default for language interpreters.
Mitigation: The `--test` flag without a file still prints usage as before.

## Rollback Strategy

All changes are additive:

- `src/repl.rs` is a new file that can be deleted entirely.
- `Interpreter::eval_item` is a new method that can be removed without
  affecting existing callers.
- `src/main.rs` changes are a small conditional branch that can be reverted
  to restore the "print usage and exit" behavior.
- `src/lib.rs` change is one line (`pub mod repl;`) that can be removed.
- No existing tests, behavior, or data structures are modified.

If rollback is needed: revert the commits. No data migration or state cleanup
required.
